# 프로그램 구조도 · 함수/변수 명세서 v7-1

[05-6-4](05-6-4_프로그램_구조도_및_함수_변수_비전_명세서.md)(v6-4) 다음 버전. **현재 실제 구현에 정합**시킨 판이다. 글 포맷은 v6-4를 유지하되, 계획 단계의 가정(Node.js 백엔드 / Next.js·Vercel 시뮬 / Neon 시뮬로그)을 **실제 구현**으로 교체한다.

**v7-1 정합 핵심(계획 v6-4 → 실제)**
- 백엔드 = **FastAPI(Python)**. (v6-4의 "Node.js 백엔드"는 미채택)
- 시뮬레이션 사이트 = **React + Vite + Express + tRPC**(로컬 dev). (v6-4의 "Next.js/Vercel" 미채택)
- 운영 DB = **MySQL 단일**. **Neon(Postgres)은 본 문서 범위에서 제외**(미사용 — 시뮬 이벤트는 백엔드 인메모리 세션 + MySQL `sim_event_log` 폴백으로 처리). Neon 분리/레거시화는 별첨 [19-11 Neon 레거시 분리 계획서·완료기록](19-11_Neon_레거시_분리_계획서_및_완료기록.md) 참조.
- 추론 = 백엔드가 **prep 번들 직접 로드**로 실시간 추론까지 수행(`python_model_loader`). 모델파트는 `POST /models/submit`로 산출물 제출.
- 과제 확장: churn(7모델) 외 **세션 바운스·다음 카테고리 추천 앙상블** 정식 편입.

기준 문서: [05-6-4](05-6-4_프로그램_구조도_및_함수_변수_비전_명세서.md) · `backend/db/schema_mysql.sql` · `backend/db/erd.dbml` · `backend/app/*`

## 0. 아키텍처 결정 원칙 (v6 계승 + v7 실제정합)

| 축 | v6-4(계획) | **v7-1 (실제)** |
| --- | --- | --- |
| 백엔드 | Node.js 서버 | **FastAPI(Python)** 단일 진입점. 봉투 응답 `{ok,data,meta,error}` + API Key |
| 운영 DB | MySQL + Neon 2-DB | **MySQL 단일**. (Neon 제외 — 시뮬 로그는 인메모리/`sim_event_log`(MySQL) 폴백) |
| 추론 주체 | 모델파트(백엔드는 저장만) | **백엔드가 prep 번들 직접 추론**(`/predict/realtime`, `/predict/diagnose`) + 모델파트 `POST /models/submit` 제출 병행 |
| 시뮬 사이트 | Next.js/Vercel + Neon | **React+Vite+Express+tRPC**(로컬), 백엔드 `/api/*` 외부계약으로 연동 |
| 전처리 분기 | registry config | prep 번들(`prep_{Model}_v2.joblib`) = pipeline+model+calibrator+feature_order+threshold 일괄 |
| 대용량 | 파일+경로 | **불변**. parquet/npz/joblib는 파일, DB엔 경로·메트릭만 |

**불변 규칙**: 스케일러/인코더/매핑은 **train에서만 fit**, 런타임은 동일 artifact로 transform만. 추론 위치는 백엔드(prep 번들), 무거운 학습은 모델파트.

### 1.3 실시간(시뮬레이션) 예측 흐름 (v7)

```
유저가 시뮬에서 상품 조회/장바구니 → FloatingChurnWidget(4초)
  → POST /api/churn/predict {session_id, user_id(=active-user), events[]}
  → sim_usecase.churn_three(): 7일(집계모델)·하자드(세션)·바운스(30분)
  → apply_policy(정책: max/ensemble/bounce_scaled/select) → churn_rate
  → action_from_events() → 쿠폰/추천/SNS 액션
  → _LAST_SIM 캐시(유저별) → 대시보드 /sim/user-score가 동일 값 표시
대시보드 개인진단 ⚡실시간 탭: st.fragment(run_every=N) 자동 갱신
```

### 1.4 입력/출력 기준 (IO 계약)
- 모든 백엔드 응답 = 봉투 `{ok, data, meta, error}` (시뮬 외부계약 `/api/*`만 raw JSON).
- 인증: `X-API-Key`(대시보드·내부), 시뮬 `/api/*`는 키 없이 CORS 허용(옵션 `SIM_SYNC_KEY`).
- 라벨: `churn`(7일), 확률 0~1, risk_level ∈ {low,medium,high} (임계 0.35/0.65).

## 2. 모델파트 → 백엔드 제출 인터페이스

### 2.1 제출 계약 `POST /models/submit`
`{ model_name, model_type(tree|linear|sequence|ensemble), preprocessing_config, artifact_path, metrics(cv/oot), predictions[] }`
→ `model_registry` 등록(+`is_active`), `model_evaluation` 적재, 예측 배치는 `prediction_log`.

- `model_registry.is_active`로 active 모델 분기. 미등록 시 per-model 산출물 중 best AUC 폴백(`eval_artifacts.all_metrics`).
- 차트는 artifact-first(`data/processed/evaluation/**`) → 폴백.

## 4. MySQL 운영 ERD (v7-1)

정본: `backend/db/schema_mysql.sql` · 시각화: `backend/db/erd.dbml`. **12 테이블**(Neon 전용 `sim_event_log`은 ERD에서 제외).

- 허브 `model_registry` ← `model_evaluation`·`prediction_log`·`recommendation`·`ensemble_member`(model_id)
- `prediction_log` ← `retention_action_log`(prediction_id)
- `ensemble_result` ← `ensemble_member`(ensemble_id)
- 피처/시퀀스/관심: `feature_user_snapshot`·`sequence_snapshot`·`user_interest`
- 얼굴: `face_user`(512d ArcFace 임베딩)·`face_login_log`
- `user_id`는 FK 미설정(가입 얼굴 ID ≠ REES46 행동 ID, 다른 공간)

## 8. 실시간 IO 계약(요약) · 배포
> 최종 IO 계약 3종(파트별)은 별첨 [19-8](19-8_최종IO계약_서버-모델팀.md)(모델) · [19-9](19-9_최종IO계약_서버-대시보드.md)(대시보드) · [19-10](19-10_최종IO계약_서버-시뮬레이션사이트.md)(시뮬) 참조.

- `POST /predict/realtime {user_id, model}` → `{churn_probability, risk_level, recommended_action, recency_days, source, ...}`
- `GET /predict/diagnose?user_id=&recency_days=` → **`{recency_days, churn:{models[],ensemble_prob,weights,weight_note}, hazard:{prob,tau_days,k}, category:{models[],weights,weight_note}, bounce:{models[],weights,weight_note}, action:{discount_pct,message,reason}|null}`** — 3 태스크 앙상블 + 가중치 설명 + 액션.
- `GET /sim/user-score?user_id=` → `{churn_rate(정책적용·단일소스), churn_7d, churn_hazard, churn_bounce, policy_mode, bounce_window_min}`.
- `GET|POST /churn-policy` → `{mode:max|ensemble|bounce_scaled|select, select_key, bounce_floor, bounce_ceiling, weights}`. **POST 시 파일 영속**(`data/processed/realtime/churn_policy.json`) → 재기동에도 유지.
- `POST /api/churn/predict {session_id,user_id,events[]}` → `{churn_probability(%), churn_breakdown[7일·하자드·바운스], recommended_action, churn_policy, source}`.
- 배포: 로컬 3프로세스(`run_all.ps1/sh`). 백엔드 단일 워커(insightface prewarm). Vercel 시뮬 ↔ 로컬 백엔드는 **Cloudflare 터널**(`VITE_FASTAPI_URL`, 26-18) 옵션.

### 8.1 앙상블 가중치 배분 (diagnose·실시간 공통)
| 태스크 | 멤버 | 가중치 | 근거 |
| --- | --- | --- | --- |
| 7일 churn | CatBoost·XGBoost·LightGBM | 균등(각 0.33) | 성능 동급 단순평균 |
| 세션 바운스 | GRU·LGBM·XGB·Cat·Transformer(5) | AUC 비례(≈0.20) | 5종 성능 유사, LogReg(타깃상이) 제외 |
| 카테고리 추천 | GRU·Transformer·LightGBM(3) | 시퀀스 0.40·0.40 / LGBM 0.20 | next-cat은 순서 중요 → 시퀀스 가중, XGB/Cat 474클래스 제외 |
> 각 블록에 `weight_note`(설명) 동반. 응답은 대시보드가 표로 렌더.

### 8.2 Churn Rate 정책 = 단일 소스 (BUG-009)
- 헤드라인은 **서버가 `apply_policy`로 한 번 계산** → 시뮬·대시보드 모두 **표시만**(로컬 재계산 금지). 정책 변경 시 양쪽 동시 갱신.

### 8.3 쿠폰 등급 · 카탈로그
- **쿠폰 등급**(기준=헤드라인 churn_rate): ≥80%→20%(긴급)·65–80%→15%(경고)·50–65%→10%(주의)·else 5%(관심). 배치 산출물 `coupon_targets.csv`(2+장바구니 채점) → `/coupons/*`.
- **카탈로그**: `/api/catalog/*` = REES46 seed **54,571 상품**(인기순). 상품/카테고리명 결측 → 화장품 품목+브랜드 매핑 라벨(26-19).
